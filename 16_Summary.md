### Before / After — 5단계 뒤 달라진 7가지

|------|----------------|----------------|

| **기준값** | 이벤트 로그에 SLA 없음 | 사용자 정의 지표로 **티켓마다 SLA 자동 할당** |

### 5단계 여정 — 각 실습이 어떤 숫자를 만들었나

이 교육은 각자 독립된 5개 실습이 아니라, **한 문장의 스토리를 5단계로 풀어낸 것**입니다. 각 실습은 이전 단계의 출력물을 입력으로 받아 한 단계씩 구체화됩니다.

#### 실습 1 — 원시 CSV를 "분석 가능한 재료"로 만들다

**한 일** : Jira에서 추출한 support_ticketing_full.csv를 이벤트 로그로 변환하고, 역할별 비용 모델 · 활동 소요 시간 · 자동화 속성(MA 필드)을 부여했으며, BPMN 참조 모델을 자동 도출해 **Should-Be 기준선**을 확보했습니다.

**결정적 결과** : 이벤트 로그가 **"건수 분석"이 아닌 "비용 분석"**이 가능한 상태가 됐고, 이후 모든 실습에서 등장하는 **$14(≈€13) 페널티 · €9,770 낭비 · €19K 절감** 같은 금액 숫자가 이 단계에서 심은 비용 모델에서 출발합니다.

#### 실습 2 — "프로세스가 느리다"를 "누가 / 어디서 / 얼마"로 바꾸다

**한 일** : Frequency · Duration · Overall Cost · Rework 4개 모델 뷰를 교차하고, BPMN 참조 모델 대비 적합성 검사를 수행했으며, 마지막으로 처방적 AI 보고서까지 자동 생성했습니다.

**결정적 결과** :
- **두 범인 활동 확정** — `Change in Request Type`(자동화율 42%)과 `Change in Person Assigned`(자동화율 0%)가 4개 뷰 모두에서 경고등
- **이탈 = 예외가 아닌 구조적 문제** — 총 3,192건 중 **이탈 2,230건(70%)**, 이탈 시 평균 **+$14(≈€13) 비용 · +7일 리드타임**
- **원인 좁혀짐(localize) 완료** — `Waiting for user`의 **63%가 Incident 유형에 집중**, 담당자 3명이 Change in person assigned의 50%+ 처리

"감"이 숫자로 바뀐 시점입니다.

#### 실습 3 — 보편적 진단을 Jira 티켓팅의 운영 언어로 번역하다

**한 일** : SLA 기준값을 사용자 정의 지표로 주입하고, 두 개의 사전 구성 대시보드(Executive Business Dashboard · Ticket Deviations)를 가져와 **시간·비용 이중 진단**을 수행했으며, 빈 캔버스에서 Annotation · Filter · Card · Elastic · Table 위젯을 조립해 **Resource Monitoring 대시보드**를 직접 구축했습니다. 마지막으로 `country_codes.csv`를 객체 테이블로 연결해 외부 마스터 데이터 결합까지 실험했습니다.

**결정적 결과** :
- **SLA 정량 증거** — High Request 티켓 해결 시간이 SLA보다 **평균 +2일 22시간** 초과
- **비용 재검증** — Total Human Cost **€113,902** 중 Changes 관련 비용 **€9,770(≈10%) = 예방 가능 낭비**
- **운영 판단 도구** — Country_1 티켓 → Resource Monitoring 3개 위젯 교차 조회 → **Assignee_2 추천**이라는 구체적 배정 결정까지 1분 내 도출

실습 2의 보편적 진단이 **현장 관리자의 일상 도구**로 성격이 바뀐 시점입니다.

#### 실습 4 — 아직 실행 안 한 개선안의 ROI를 미리 계산하다

**한 일** : Change in Person Assigned를 **근본 원인 제거(증상 완화가 아닌)** 방식으로 해결하기 위해 `First Assigned` 자동 할당 시나리오를 설계하고, **자동화율 80% · 봇 1개 · 처리 시간 1분**으로 What-if 시뮬레이션을 실행했습니다. 도착 분포·직원 가용성·활동 통계는 실제 로그에서 자동 추출돼 **현실 패턴의 연장선**에서 예측.

**결정적 결과** :
- **6개월 3,200건 → 연간 6,400건 × 건당 €2.99 절감 = 연간 ≈ €19,000**
- **자동화 구현 비용 €15,000 → 첫해 내 회수 → 2년차부터 €19,000/년 전액 순이익**
- **"자동화하면 좋을 것 같다"가 "첫해 회수 · 2년차 순익 전환"이라는 경영 언어로 번역**

프로세스 마이닝이 **분석 도구에서 투자 의사결정 도구로 진화**한 시점입니다.

#### 실습 5 — 자동화가 배포되기 전의 오늘도 SLA를 지키다

**한 일** : 레거시 `Resolution SLA` 대시보드를 import한 뒤 2.x 객체 모델에 맞춰 **4가지 교정**(오탈자·데이터 소스·필드 한정·Keep last)을 적용하고, `Keep only running cases` 필터로 대시보드의 시간축을 **과거 평균에서 현재 감시로 뒤집었으며**, 모니터-서비스 연결로 **Slack 자동 호출 체계**를 이해했습니다.

**결정적 결과** :
- **실행 중 47개 티켓** 중 High 우선순위 7건 평균 개방 **18일 → SLA 2일의 9배 초과** 식별
- **대시보드의 시간축을 필터 한 줄로 전환** — 실습 3의 IS_RUNNING = 0 전략 분석 뷰가 IS_RUNNING = 1 운영 감시 뷰로 변신
- **보고 → 인지 → 대응**이 **알림 → 클릭 → 대응**으로 전환

자동화 완성을 기다리지 않는 **첫날부터 가동되는 단기 개선** 수단을 확보한 시점입니다.

### 재사용 가능한 산출물 지도

교육이 끝난 지금 여러분은 **세 범주의 산출물**을 가지고 있습니다. 이 산출물들은 Jira 티켓팅을 넘어 다른 프로세스에도 동일하게 적용할 수 있는 **재사용 가능한 템플릿**입니다.

#### 분석 산출물 (실습 1·2)

- **이벤트 로그 · 비용 모델 · BPMN 참조 모델** — 모든 분석의 재료
- **Frequency · Duration · Cost · Rework 4-렌즈 교차 진단 방법론**
- **처방적 AI 보고서** — 수일의 분석을 1분에 압축

#### 운영 산출물 (실습 3·5)

- **5개 대시보드** — Executive · Ticket Deviations · Resource Monitoring · Automation simulation · Resolution SLA
- **사용자 정의 지표** — 이벤트 로그에 없는 비즈니스 규칙(SLA 등)을 주입하는 메커니즘
- **객체 테이블** — 외부 마스터 데이터(국가·벤더·자재)를 분석 범위에 붙이는 메커니즘
- **모니터 + 서비스** — 위반을 Slack·이메일·IBM AppConnect로 자동 호출하는 체계

#### 경영 산출물 (실습 4)

- **What-if 시뮬레이션 프레임** — 실제 로그 패턴 기반 조건부 예측
- **€19K/년 ROI 비즈니스 케이스** — 경영진 설득용 정량 근거
- **근본 원인 제거 vs 증상 완화 판단 프레임** — 개선안 선택 기준

### 프로세스 마이닝 단계

다섯 단계는 바꿔 끼울 수 없습니다. **앞 단계의 출력이 다음 단계의 입력**이기 때문입니다.

- 비용 모델(1)이 없으면 "얼마 낭비"(2)를 말할 수 없음
- 두 Change 활동 확정(2)이 없으면 "무엇을 모니터할지"(3·5)·"무엇을 시뮬레이션할지"(4)가 정해지지 않음
- SLA 기준값·Resource Monitoring 대시보드(3)가 없으면 운영 감시(5)의 재료가 없음
- ROI(4)가 없으면 경영진이 자동화 투자를 승인하지 않음
- 모니터(5)가 없으면 분석 결과는 **사람이 안 열면 무의미한 화면**에 머무름

<strong>"데이터 입력 → 현실 진단 → 운영화 → 개선 예측 → 현장 호출"</strong> — 이 순서가 분석을 조직 행동으로 연결하는 유일한 경로입니다.

### 실전 적용 — 다음 프로세스에 이 패턴을 옮기려면

Jira 티켓팅 대신 여러분 조직의 프로세스(구매 · 송장 · 보험 청구 · 온보딩 등)에 같은 사이클을 적용할 때의 체크리스트.

1. **이벤트 로그를 비용으로 환산 가능한 상태로** — 역할별 비용 · 활동 시간 · 자동화 속성을 먼저 주입(실습 1 패턴)
2. **4-렌즈 + 참조 모델 + 처방적 보고서로 범인 확정** — 교차 분석 없이 한 뷰만 보면 오판(실습 2 패턴)
3. **SLA·비즈니스 규칙은 사용자 정의 지표로 주입** — 이벤트 로그 수정 없이 기준값을 분석에 포함(실습 3 패턴)
4. **근본 원인 제거안을 골라 historical 시뮬레이션으로 ROI 산출** — 증상 완화가 아닌 구조 변경을 우선(실습 4 패턴)
5. **자동화 완성을 기다리지 말고 운영 모니터부터 켜기** — 데이터 파이프라인이 실시간이면 첫날부터 가동(실습 5 패턴)

### IBM Process Mining이 제공한 핵심 가치

이 교육 전반에 걸쳐 반복 확인된 6가지 가치.

**데이터 기반 진단** — 직관·경험이 아니라 이벤트 로그에서 병목·재작업·비용 낭비를 **객관적으로** 추출. 이 실습에서는 두 Change 활동이 리드타임의 74%를 차지한다는 사실이 데이터로 증명됐습니다.

**분석 속도** — 수일 걸리던 패턴 발견을 **처방적 AI 보고서가 1분에 압축**. 분석가의 결론을 AI가 자동 재확인하여 **분석 품질 교차 검증**까지 한 번에 달성.

**유연한 확장** — 이벤트 로그에 없는 외부 데이터도 **사용자 정의 지표(SLA)**와 **객체 테이블(국가 코드)**로 즉시 결합. 데이터 파이프라인을 다시 만들지 않고도 분석 범위가 수평 확장됩니다.

**투자 타당성 정량화** — What-if 시뮬레이션이 **실제 로그 패턴의 연장선**에서 돌아가기에 예측 결과가 가정이 아닌 조건부 현실 — €19K/년·첫해 회수 같은 경영 언어로 투자 설득.

**즉시 실행성** — 장기 자동화를 기다리지 않고 **운영 모니터로 첫날 단기 성과**를 확보. "인사이트 → 액션"이 24시간 내로 완성되는 구조.

**전체 수명 주기 지원** — 수집 → 분석 → 설계 → 시뮬레이션 → 운영이 **하나의 플랫폼**에서 이어지므로, 일회성 분석이 아닌 **지속적 프로세스 최적화 문화**를 조직에 심을 수 있습니다.

### 마지막 한 문장

> **프로세스 마이닝은 "분석 도구"가 아니라 "조직을 움직이는 언어"입니다.**
>
> 숫자가 없으면 설득이 안 되고, 설득이 없으면 변화가 없고, 변화가 없으면 분석은 그저 예쁜 리포트로 끝납니다. 이번 교육에서 여러분이 손에 쥔 것은 **CSV 한 장을 €19,000 ROI와 Slack 알림으로 바꾸는 5단계 번역 기술** — 이 기술을 여러분 조직의 다음 프로세스에 그대로 옮겨 적용해 보십시오.
